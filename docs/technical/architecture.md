# Архитектура

Бизнес-правила находятся в `core`: сервисы, модели, RBAC, фичтоглы, порты и
интеграции не зависят от Telegram. Текущий интерфейс расположен в каталоге
`core/interfaces/telegram/`; новые интерфейсы должны вызывать те же сервисы и
реализовывать контракты из `core/ports.py`.

Направление зависимостей: `interface -> application service -> database /
integration ports`. Ядро не должно импортировать aiogram. `core/application.py`
оставлен только как фасад совместимости для старых импортов.

База данных создаётся в `core/runtime.py` через фабрику драйверов. SQLite
реализован в `core/integrations/database/`; поэтому смена хранилища не требует
изменений Telegram-интерфейса или прикладных сервисов.

## Фичтоглы

Все поддерживаемые ключи перечислены в `config/features.config`. Зависимости объявлены
в `core/features.py` и при `STRICT_FEATURE_DEPENDENCIES=true` проверяются при
старте. Включить дочернюю фичу при выключенной родительской нельзя — приложение
завершится с понятной ошибкой. Флаг строгого режима оставлен отключаемым для
совместимости со старыми частичными конфигурациями.

## Права

Роли, наследование и permissions задаются в `config/permissions.json`.
`team_lead` является вычисляемой ролью поверх сотрудника, владелец имеет `*`.
Проверки новых сценариев следует выполнять через `PermissionPolicy.allows`.

## Секреты интеграций

Пароли хранятся в SQLite только как Fernet ciphertext. Мастер-ключ задаётся
переменной `INTEGRATION_SECRET_KEY` и не должен попадать в репозиторий или БД:

```powershell
.\.venv\Scripts\python.exe -c "from cryptography.fernet import Fernet; print(Fernet.generate_key().decode())"
```

Для Яндекс Почты и Mail.ru рекомендуется создавать отдельный пароль приложения.
При отключении интеграции зашифрованный секрет удаляется.

Поддерживаемые почтовые провайдеры: Google, Microsoft 365, Яндекс, Mail.ru и
произвольный SMTP. SMTP-адаптер находится в `core/integrations/mail.py`.
